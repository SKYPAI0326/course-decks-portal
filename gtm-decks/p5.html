5-2 · Dedup Soul
11 / 22 講義 · 5-2
Concept · 同一筆轉換不能算兩次

Dedup 的靈魂：event_id

同一筆 purchase，Pixel 跟 CAPI 各送一份，
Meta 收到會被算成兩筆—— ROAS 立刻假。

兩邊都帶同一個 event_id（通常用 order_id），
Meta 看到重複只算一次。
Event Match Quality 目標 ≥ 7.0 才算健康
event_id
order_id（最穩，自然唯一且可追溯）
Recommended
event_time
Pixel 與 CAPI 兩邊差距 < 30 秒才算同事件
Window

5-2 · Three Paths
12 / 22 講義 · 5-2
Decision · 依工程資源挑一條

三條實作路徑：選你能養的

Path A · 入門
Zapier · Make
WooCommerce → Webhook → Make → CAPI。
免寫 code，但延遲 30 秒+，月有事件量上限。
Path B · 進階
sGTM
Cloud Run 跑伺服器端 GTM，所有事件先進 sGTM 再分流。
低延遲、要會 GCP，月成本 $15–40。
Path C · 專業
直串 CAPI
後端在訂單成立那刻 POST 到 graph.facebook.com。
最穩最快，需要工程資源。

本課演練採 Path A · Pixel + Zapier Webhook—— 90% 中小電商的可行解。

5-2 · User Data Hash
13 / 22 講義 · 5-2
Concept · Hash 不是為了資安，是為了比對

user_data 一定要 SHA-256

Meta 拿你送的 hash 去跟他們資料庫比對，
找出「這個人是哪個 FB 帳號」—— 比對到了 EMQ 才會高。

em · email
小寫
去空白
normalize 後再 hash，否則比對失敗
ph · phone
E.164
+886…
不要 0 開頭、不要破折號
fn / ln · name
小寫
去重音
姓 / 名分開欄位送

Meta 事件管理工具 → Test Events 看 EMQ 分數，< 5.0 通常就是 normalize 沒做。
